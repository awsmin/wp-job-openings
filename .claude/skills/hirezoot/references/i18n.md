# Internationalization (i18n)

Use this file for any user-facing string. This repo's `phpcs.xml` `WordPress.WP.I18n` sniff allows
only **two** text domains: `default` and `wp-job-openings` (see `coding-standards.md`) — simpler
than the sibling Pro Pack repo, which has to juggle two plugin domains. Every user-facing string in
this repo should use `wp-job-openings` (or rely on WordPress core's `default` domain for genuinely
core-owned strings) — there's no legitimate reason to reach for another plugin's text domain here,
even when the string is related to a sibling add-on (e.g. an "upgrade to Pro" notice should still
translate under `wp-job-openings`, since it's this plugin's own UI making the offer).

## Translator comments and plurals — already used correctly, keep the pattern

The codebase already has 57 `translators:` comments, consistently placed immediately above the
`sprintf`/`printf`/`_n` call whose placeholder they document:

```php
/* translators: %1$s: main plugin, %2$s: current plugin, %3$s: minimum required version of the main plugin, %4$s: plugin update link */
```

- Add a `translators:` comment above any new `sprintf`/`printf` call with a placeholder whose
  meaning isn't obvious from context alone.
- Use `_n()` for count-dependent strings, matching the ~11 existing call sites (e.g.
  `_n( '%s job listing updated.', '%s job listings updated.', $bulk_counts['updated'], 'wp-job-openings' )`)
  — never string-concatenate a plural suffix. `_nx()` isn't used anywhere in this repo; there's no
  need to introduce it unless a genuine disambiguation-by-context case comes up.

## Translation functions

- Every user-facing string wrapped in `__()`, `_e()`, `esc_html__()`, `esc_attr__()`, `_n()`, or
  `_x()` with the `wp-job-openings` domain.
- Never concatenate translated strings or interpolate variables directly into a translatable string
  — use `sprintf`/`printf` with `%s`/`%d` placeholders, per the pattern above.

## WPML integration is narrow — don't assume it covers more than it does

`inc/translation/class-awsm-job-openings-wpml.php` does exactly one thing: whenever the
`awsm_jobs_filter` option is saved, it registers the labels of dynamically-created specification
taxonomy terms (which live in an option, not in code strings a build-time scanner would find) as
WPML translatable strings, via `do_action( 'wpml_register_single_string', ... )`. It does **not**
touch CPT translation support, post content, or any other multilingual concern. If a new feature
introduces another **dynamically-generated, options-stored** label that needs WPML translatability
(the same category of problem this class solves), extend it with the same
`wpml_register_single_string` pattern on the relevant `update_option_{name}` hook — don't assume
WPML already picks it up just because other strings in the plugin are translatable via the normal
`.pot`/`.po` mechanism.

## POT file and regeneration

`languages/wp-job-openings.pot` currently has 527 `msgid` entries. It's regenerated by the `translate`
gulp task (`gulp-wp-pot` scanning `./**/*.php`, excluding `build/` and `vendor/`) — run
`npm run translate` after adding/changing translatable strings if you need the `.pot` file itself
updated as part of the task (not every string-level change requires this; check whether the task
actually calls for a `.pot` regeneration before running it).
